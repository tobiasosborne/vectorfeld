# 2026-04-27 — vectorfeld-eb0 shipped (graft Phase 3 in-place source-font edits)

Closes the visible seam in the graft modifications path. Until today,
modifying a source-PDF text element (recolor, attribute edit) routed
through graft and overlaid the result in Carlito — visibly different
from the surrounding source text in any non-Carlito source. The eb0
work makes the engine extract the source's embedded font, register
it on the output page, and re-emit the modified run in its original
typeface. Result: edited text blends invisibly with un-edited
surroundings.

5 sub-beads of `vectorfeld-eb0` shipped today, after the parent
`vectorfeld-yyj` track wrapped earlier in the same session
(see [2026-04-27-yyj-shipped.md](2026-04-27-yyj-shipped.md)).

## The single new failure mode: subset coverage

Source PDFs ship subsetted fonts. The 6-letter-prefix Calibri-Bold
in the flyer fixture only contains glyphs the original PDF
rendered. A user who recolors an existing Calibri heading is
covered (every char already on the page); a user who EDITS the
heading to add a glyph the subset doesn't contain (e.g., a Greek
letter, a math symbol) would emit .notdef boxes if we didn't
guard.

Fix: `fontCoversText(font, text)` in graftCs.ts checks
`font.glyphForCodePoint(cp).id === 0` before each run is emitted.
On a miss, the run's fontKey swaps to the registry's fallback key
(Carlito). The user sees Carlito instead of Calibri for that
specific run; everything else stays in Calibri. Better than
.notdef boxes; gracefully degrades.

This split-mid-headline behavior is technically a (different)
seam, but it only fires for chars the source PDF physically
doesn't have a glyph for — which the graft engine literally cannot
render in the source font. The visible result is "this character
came from a different font" rather than "this character is a
square box."
