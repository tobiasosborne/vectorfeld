## Two architectural calls worth keeping

### 1. Stories 06 / 10 mutate the DOM directly

The natural UI flow — click the headline text → press Delete (06) or
change fill via the panel (10) — fails because:

- MuPDF wraps each imported text run in a `<g>` container.
- `selectTool.hitTestElement` only walks layer-direct children, so a
  click on the headline selects the wrapping `<g>`, not the inner
  `<text>`.
- The PropertiesPanel's fill picker only mounts when the SELECTED
  element has a solid fill — `<g>` wrappers have no `fill` attribute,
  so `detectFillType` returns `'none'` and the picker stays unmounted.
  Changing the fill-type select to `'solid'` distributes the new fill
  to children, but the panel still doesn't see a fillable selection so
  the picker still doesn't mount → `h.setFill` throws.

Pragmatic gate strategy: stories 06 and 10 use `page.evaluate` to
`element.remove()` / `element.setAttribute('fill', '#ff0000')`
directly. Both stories include explicit verification (count drop /
fill-attribute check) so a silent no-op fails the recording phase.

The gate's purpose is byte-match invariance on export, which only
depends on the FINAL DOM state. Filed `vectorfeld-qj7` (P3) to expose
a `window.__vfTest` hook so these stories can later switch to a
UI-driven path without committing to a particular group-drill-in UX.

### 2. Story 09 uses a custom mm-space drag

The pen tool's "smooth handle" gesture is a real mouse drag from the
new anchor (down → move → up), not a click sequence. Story 09 inlines
a `dragAtMm(page, from, to)` helper that converts mm coords to screen
pixels via `svg.getScreenCTM()` and runs the down/move/up dance with
intermediate move events so the tool's RAF-driven preview gets ticks.
Coordinates stay in document mm so the geometry is independent of
canvas zoom and pixel ratio.

If a third story ever needs this, promote it to `harness.mjs`.

## TDD findings worth keeping

- **Imported PDFs split text by font/style run.** The flyer headline
  "Kurzfristige Hilfe bei englischen Vorträgen" is split across
  three `<text>` elements: one for the dropcap "K", one for the
  remaining "urzfristige Hilfe…", one for trailing whitespace. A
  literal substring match on "Kurzfristige" finds nothing — the
  `<text>` content is `"urzfristige Hilfe…"`. Stories 06/10 pick the
  largest-font multi-char element instead, which resolves uniquely
  (only the headline runs at 24.96pt).

- **`pdftotext` output matches the rendered text but not the
  serialized SVG structure.** `pdftotext` extracts logical text;
  MuPDF's text=text mode preserves source-stream structure including
  per-glyph splits. When debugging a "text not found" failure, probe
  the rendered SVG, not the source PDF.

- **Story 06's silent no-op trap.** First version of story 06 did
  `waitForFunction((id) => !document.getElementById(id), target.id)`
  to confirm deletion. But MuPDF-imported elements have no `id`
  attribute (vectorfeld assigns ids only on user-added elements), so
  `target.id` was `null` and `document.getElementById(null)` is
  always falsy → the wait passed even when nothing was deleted.
  Replaced with a count-based assertion (`afterCount === beforeCount
  - 1`) so silent no-ops fail the recording phase.
